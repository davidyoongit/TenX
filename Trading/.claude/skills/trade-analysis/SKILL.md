## 로그 구조

`trades/YYYYMMDD.json`:
```json
{
  "date": "YYYYMMDD",
  "strategy": "전략명(파라미터)",
  "trades": [
    {
      "code": "069500",
      "name": "KODEX 200",
      "action": "buy" | "sell",
      "qty": 10,
      "price": 32000,
      "time": "09:15:30",
      "buy_price": 31500,   // sell만
      "pnl_amt": 5000,      // sell만
      "pnl_pct": 1.59,      // sell만
      "reason": "eod" | "tp" | "sl"  // sell만
    }
  ],
  "summary": {
    "strategy": "...",
    "trade_count": 5,
    "win": 3, "lose": 2, "win_rate": 60.0,
    "total_pnl_amt": 150000,
    "total_pnl_pct": 1.5,
    "best_trade": {...},
    "worst_trade": {...}
  }
}
```

## 분석 절차

1. `trades/{날짜}.json` 파일을 Read
2. `trades` 배열에서 `action == "sell"` 항목만 추출
3. 아래 4개 차원 분석 수행
4. 각 차원의 결과를 전략 수정 권고로 변환
5. `_workspace/04_trade_analysis.md`에 저장

## 4개 분석 차원

### 차원 1: 매도 이유 분포
| reason | 의미 | 수정 방향 |
|--------|------|---------|
| `sl` 비율 > 40% | 손절 기준이 너무 타이트 | `stop_loss_pct` 완화 (더 음수로) |
| `tp` 건수 = 0 | 익절 목표가 너무 높거나 추세 부족 | `take_profit_pct` 하향 검토 |
| `eod` 100% | 장중 조건이 한 번도 발동 안 됨 | 조건 threshold 재검토 |

### 차원 2: 손익률 분포
- 손절 평균 `pnl_pct` vs `stop_loss_pct` 설정값 차이가 0.5%p 이상이면 괴리 있음
- 전체 손익이 worst_trade 하나에 크게 의존하면 집중 리스크 경고

### 차원 3: 진입 타이밍
- 매수 `time`을 3구간으로 분류: 장초(~10:00), 오전(10:00~12:00), 오후(12:00~)
- 특정 구간의 평균 손익이 현저히 낮으면 해당 시간대 진입 제한 권고

### 차원 4: 종목 집중도
- 손절 건이 특정 code에 2건 이상 집중 → 해당 종목 제외 또는 조건 강화 권고
- 매수 후 30분 이내 손절 건 비율 > 50% → 초기 변동성 대기 조건 추가 권고

## 권고 작성 기준

- 데이터 근거 필수: 건수·비율·수치를 반드시 제시
- 파라미터명 명시: "손절 완화" ❌ → "`stop_loss_pct` -1.5% → -2.0%" ✅
- 거래 건수 < 3이면 권고 생략, "데이터 부족" 명시
- 권고는 최대 5개, 우선순위 순으로 정렬
